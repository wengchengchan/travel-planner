# Travel Planner Trigger Spec

## 目標

當使用者丟出類似「幫我規劃旅遊」的指令時，不要每次重新判斷流程，而是自動展開固定工作流。

---

## 1. 觸發語

以下語句可視為進入 `travel-planner` 工作流的觸發：

- 幫我規劃旅遊
- 幫我排旅遊行程
- 幫我安排旅行
- 幫我做一份行程
- 幫我做旅遊規劃
- 幫我排 3 天 2 夜行程
- 我想去東京 / 北京 / 首爾，幫我排一下

核心判斷：
- 使用者要的是「旅遊地點 + 天數 + 行程安排」
- 不是單純問景點知識，而是要實際規劃

---

## 2. 自動展開流程

一旦命中觸發語，預設自動進入以下流程：

### Step A：收集基本資料
用問答方式補齊：
- 地點
- 幾天幾夜
- 幾個人
- 重點景點
- 旅遊風格
- 不想要什麼
- 幾點出門
- 行程密度
- 最後一天是否回程
- 預算
- 想吃什麼
- 住宿區域是否已知

### Step B：產出第一版自然語言行程
- 先給 Day 1 ~ Day N 的安排
- 解釋為什麼這樣排
- 讓使用者先校正方向

### Step C：整理成資料底稿
- 產出 `trip-<city>-draft.json`
- 作為後續頁面與資料流基底

### Step D：視需要產出頁面內容
- 產出網頁文案稿
- 或直接生成 `<city>-YYYY-MM.html`

### Step E：視需要更新首頁
- 若要納入正式展示站，再更新 `index.html`

---

## 3. 自動做 vs 需要確認

### 可預設自動做
- 問答收集資料
- 第一版行程安排
- trip draft JSON
- HTML 草稿生成
- 首頁更新草稿

### 建議先確認再做
- 覆蓋既有正式頁面
- 刪除中間產物
- `git commit`
- `git push`

原則：
- 對話中的規劃流程可自動展開
- 寫入 repo / 上版屬於第二層，最好一句確認後再做

---

## 4. 預設回應模式

當命中觸發語，回應應直接切入資料收集，而不是先講很多方法論。

建議開場：
- 先收幾個關鍵資料，我幫你排第一版。
- 直接問：去哪裡、幾天幾夜、幾個人？

原則：
- 先做，不先講太多
- 問題分批問，不一次塞太多
- 每次收一組，降低使用者負擔

---

## 5. 最小成功條件

只要做到以下幾點，就算這條觸發流程成立：
- 使用者一句話觸發
- 自動進入問答收集
- 自動收成第一版行程
- 可延伸到 JSON / HTML / 首頁更新

---

## 6. 一句話總結

`travel-planner` 的理想狀態不是「使用者叫我填表」，而是：

**使用者只要說想去哪，我就自動進入旅遊規劃工作流。**
